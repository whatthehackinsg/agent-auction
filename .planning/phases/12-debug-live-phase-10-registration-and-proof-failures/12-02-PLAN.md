Only call `verifyMembershipProof()` once the expected on-chain privacy state is known.
4. Once on-chain privacy state is available, pass it into `verifyMembershipProof()` and convert invalid results into `StructuredActionError` payloads that preserve fail-closed rejection while surfacing the actual mismatch cause.
5. In `mcp-server/src/tools/join.ts`, stop flattening every membership-proof rejection into the same generic `PROOF_INVALID` guidance. Parse the engine's structured error payload when present and map it to likely causes and exact next actions, such as:
   - wrong `agentId` / `agentStateFile` pairing
   - stale or mismatched local proof state
   - on-chain privacy state not yet present or unreadable for the target agent
   - privacy registration visible to `check_identity` but not consistent with the local saved state
6. Keep `proofPayload` as an advanced override only. Do not make manual override the default path, and do not relax proof enforcement to push the join through.
7. The resulting diagnostics or summary must make the readiness boundary explicit: if `check_identity` was green but JOIN still failed, the user should be told that identity/privacy visibility passed while proof-state alignment did not.
