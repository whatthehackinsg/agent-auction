Extend `mcp-server/src/tools/bond.ts` so it registers a new `deposit_bond` tool and keeps `post_bond` as advanced/manual fallback.

Implementation requirements:
1. `deposit_bond` should default to the live deployed escrow flow, not an imaginary `deposit()` contract call. The current deployed system is direct USDC transfer to escrow followed by engine receipt verification. Follow the real contract path from `agent-client/src/auction.ts` and `engine/src/index.ts`.
   - Do not add a no-op `approve()` step just to mirror stale wording. If allowance-based deposit is desired later, that is a contract/API change outside this phase.

2. Preflight before sending funds:
   - fetch auction details and resolve the required deposit amount automatically
   - check current bond status / existing record for the target agent
   - if current state is already `PENDING` or `CONFIRMED`, return that state idempotently

3. On a fresh bond:
   - perform the on-chain USDC transfer to `AuctionEscrow`
   - POST the tx hash to `/auctions/:id/bonds`
   - do a short smart-wait poll for confirmation; if it lands fast, return `CONFIRMED`, otherwise return `PENDING` with tx hash and status instructions

4. Default the transfer signer to the main owner wallet, but allow the advanced funding-wallet override from Task 1. Preflight the selected signer against the engine's identity binding rules so mismatch errors are surfaced before sending funds.

5. Preserve `post_bond` as the advanced/manual recovery tool for operators and already-submitted transactions. Do not remove it from the server or docs.

6. Keep responses actionable. The normal path should tell the agent what to do next (`join_auction`) and the pending path should tell it how to re-check status.
