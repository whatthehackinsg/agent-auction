Implementation requirements:
1. Build a metadata JSON object for the new identity and encode it as a `data:` URI for the ERC-8004 `register(string agentURI)` call. Use Claude's discretion for exact metadata fields, but include enough structure that the payload is self-describing.

2. Follow the live registration flow from `agent-client/src/identity.ts`: write the ERC-8004 transaction, wait for receipt, parse `Registered`, and trust the emitted `agentId` instead of assuming a local counter.

3. Immediately bootstrap privacy membership in the same tool call:
   - call `prepareOnboarding(...)`
   - call `registerOnChain(...)`
   - persist the returned private state using the new agent-state helper
   This is required by the locked "full bootstrap" decision; do not ship a mint-only tool.

4. After both on-chain steps succeed, call the existing engine `/verify-identity` path (same contract used by `check_identity`) and include the readiness payload in the tool response.

5. Response shape must include at minimum:
   - `agentId`
   - ERC-8004 tx hash
   - privacy registration tx hash
   - saved state-file path
   - resolved wallet
   - readiness / missing-step summary

6. Return clear fail-closed errors if any step is missing prerequisites or if the second on-chain step fails after minting. If partial success occurs, surface the minted `agentId` and tx hash so recovery is possible.
