<objective>
Add `bidCommitment` field to `AuctionEvent` and thread it from the BidRange proof verification result through `ValidationMutation` → `ingestAction` → event storage. This makes the event log self-describing for bid proofs, matching the existing `zkNullifier` pattern for membership proofs.

Purpose: Without this, MCPE-02's success criterion ("produces an AuctionEvent with bidCommitment populated") cannot be satisfied. The engine extracts `bidCommitment` from proof verification but currently discards it.

Output: Three small edits to engine types, handler, and auction room — all following the established `zkNullifier` threading pattern.
</objective>

<execution_context>
This plan has NO dependencies — it modifies only engine code that exists today. It can run in parallel with Plan 02-01 (MCP server infra) and Plan 02-02 (tool wiring). The changes follow the exact same pattern as `zkNullifier`: optional field on `ValidationMutation`, threaded to `ingestAction`, stored in `AuctionEvent`.
</execution_context>
