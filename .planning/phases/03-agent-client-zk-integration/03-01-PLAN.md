### Engine nullifier coupling contract (READ-ONLY — DO NOT MODIFY engine code)

When a ZK membership proof is present, the engine:
1. Extracts `nullifier = publicSignals[MEMBERSHIP_SIGNALS.NULLIFIER]` (index 2, string)
2. Converts to `nullifierBigInt = BigInt(nullifier)`
3. Passes `nullifierBigInt` into EIP-712 `verifySignature()` as the `Join.nullifier` field

The agent-client signer MUST use the SAME Poseidon nullifier value when signing — otherwise the engine will reject with `Invalid EIP-712 signature`.

### Agent state file format (test-agents/agent-1.json)

All bigint fields use trailing `n` convention: `"48522356...n"` → strip `n`, pass to `BigInt()`.
```json
{
  "agentId": "1n",
  "agentSecret": "48522356...n",
  "salt": "93316576...n",
  "capabilities": [{ "capabilityId": "1n" }],
  "leafHashes": ["5355868...n"],
  "capabilityMerkleRoot": "14499290...n",
  "registrationCommit": "0x47b3e1..."
}
```
The `usedNullifiers` field does not exist in current files — `loadAgentState()` must default to `[]`.

### Known project patterns

- `buildPoseidonMerkleTree()` returns `{ root, layers }` in TypeScript type, but actual object also includes `zeroHashes`. Cast as `(treeResult as any).zeroHashes` — established pattern.
- BidRange `maxBudget=0` means uncapped auction → substitute `BigInt(2 ** 48)` sentinel (circuit constraint requires non-zero maxBudget).
- `ethers` is needed by `readRegistryRoot()` from `@agent-auction/crypto` — it comes as a transitive dependency via the crypto package.

### Anti-patterns to avoid

- **Do NOT hardcode signal index 2** — always import `MEMBERSHIP_SIGNALS.NULLIFIER` from `@agent-auction/crypto`.
- **Do NOT persist nullifier before engine call succeeds** — write to `usedNullifiers` AFTER successful engine response only.
- **Do NOT import snarkjs directly** — use `generateMembershipProof()` and `generateBidRangeProof()` from `@agent-auction/crypto` which handle wasm/zkey path resolution.
- **Do NOT generate BidRange proof before local validation** — pre-validate with `validateBidRange()` first to avoid wasted ~2s fullProve.

          : `bid ${bid} > maxBudget ${maxBudget}`
        super(
          'BID_OUT_OF_RANGE',
          `Bid violates range constraint: ${constraint}`,
          `Bid must satisfy reservePrice (${reservePrice}) <= bid <= maxBudget (${maxBudget})`,
