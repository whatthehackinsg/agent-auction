<objective>
Thread ZK proof fields (zkNullifier, bidCommitment) through the engine WebSocket broadcast and extend the frontend AuctionEvent interface to receive them. This is the data plumbing prerequisite — without it, ZK badges in the activity feed would never render because the fields are silently dropped before reaching the browser.

Purpose: Enable ZK proof data to flow from engine Durable Object storage through WebSocket to the frontend React state, unblocking FRNT-01 (ZK badges) and FRNT-02 (nullifier indicators).
Output: Engine broadcastEvent updated with ZK fields; frontend AuctionEvent interface extended with optional ZK fields.
</objective>

<execution_context>
This is Phase 4 Plan 1 — data plumbing. Phases 1-3 completed all ZK proof generation, engine verification, and agent-client integration. The engine already stores zkNullifier and bidCommitment in DO storage and D1 event payload — but the broadcastEvent method at line 1124 silently drops them because its parameter type doesn't include these fields, and the call site at line ~658 doesn't pass them.

The fix is small and surgical: add two optional fields to broadcastEvent's parameter type, forward them in maskedEvent construction, and pass them from the ingestAction call site. On the frontend side, add two optional fields to the AuctionEvent interface — the existing spread pattern in onmessage already passes unknown fields through.

**Critical safety note:** zkNullifier and bidCommitment are cryptographic hash outputs (Poseidon nullifier, bid commitment). They are NOT identity-revealing — the agentId is already masked separately. Including them in public broadcast adds cryptographic evidence without deanonymizing agents.
</execution_context>
