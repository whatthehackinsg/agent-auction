<objective>
Inject ZK proof verification badges, nullifier tags, and privacy explainer panels into the spectator UI. This is the visual display layer that makes the cryptographic privacy stack visible to hackathon judges. Gold badges with truncated hashes appear on ZK-verified events in the activity feed; static explainer panels explain the proof system in both the auction room and agent profile pages.

Purpose: Judges can visually confirm ZK proof verification is working — gold badges on JOIN/BID events prove real Groth16 proofs were submitted, and explainer panels provide context on what the cryptographic primitives mean.
Output: Activity feed with ZK badges + nullifier/commitment tags; auction room ZK explainer panel; agent profile ZK glossary panel.

**DEFERRED (not implemented):** DEMO-01 (full E2E Base Sepolia live run) and DEMO-02 (CCIP narrative) are deferred per user decision. They will be recorded when all work is complete.
</objective>

<execution_context>
This is Phase 4 Plan 2 — visual display. Plan 01 completed the data plumbing: engine broadcastEvent now threads zkNullifier/bidCommitment through WebSocket, and the frontend AuctionEvent interface includes these fields. Events with ZK proofs now arrive at the browser with `zkNullifier` (JOIN events) and `bidCommitment` (BID events) populated.

The task is pure composition of existing UI primitives:
- `Badge variant="warn"` produces gold text with amber glow (the ZK visual identity)
- `truncateHex()` produces `0xab12...f9c3` format
- `PixelPanel accent="gold"` produces the established gold panel with `#d7aa61` border

**Key design decisions from CONTEXT.md:**
- Badge color: gold/amber (variant="warn")
- Nullifier is a SEPARATE element from the badge (two distinct crypto evidences)
- Non-ZK events: no "UNVERIFIED" label (clean, no clutter)
- Explainer tone: Technical with labels (real terms + one-line plain-language)
- Agent profile: Replace placeholder with circuit specs (no live per-agent data — D1 doesn't store ZK fields)

**Pitfall to avoid:** `bidCommitment === '0'` is the sentinel for non-ZK bids. Guard with `e.bidCommitment && e.bidCommitment !== '0'` before rendering.
</execution_context>
