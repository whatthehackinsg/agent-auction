## Level 1：可验证交付（Machine-verifiable delivery）（P0 首选）

**特征**

- 输入/输出能结构化描述
- 验收能自动化（确定规则：通过/失败或可量化评分）
- 通常不需要高权限，不涉及敏感数据
- 最适合用 **escrow 自动释放**（通过即付款）

**典型对象（类别 + 例子）**

1. **代码/工程交付（强验证）**
    - 提交 PR / patch，CI 测试全绿；性能指标达标；lint/静态检查通过
2. **结构化数据交付**
    - 把网页/文档抽取成指定 schema（JSON/CSV），通过 schema validator + 抽样对比
3. **可复现计算/推理任务**
    - 给定数据集+脚本，输出结果 hash 与评测分数（比如基准任务跑分、模型推理产物）
4. **链上/可证明动作**
    - 提交指定 on-chain 交易、生成证明、返回 tx hash / receipt（验收可自动查链）
5. **API 结果型任务（强约束）**
    - 以明确接口返回为验收（例如“返回某系统中满足条件的记录列表”，可对比一致性）

**适合的拍卖机制**

- reverse/scoring auction（价格 + 时延 + 成功率），sealed-bid 也很自然

## Level 2：半可验证交付（Semi-verifiable / Human-in-the-loop）

**特征**

- 有客观指标，但质量仍有主观或概率成分
- 需要人类抽检/复核或多轮迭代
- 适合 **里程碑付款 + 质检抽样 + 争议流程**

**典型对象**

1. **研究/报告/调研**
- 有结构要求与引用要求，但“好不好”需要评审
1. **内容生成（营销文案、产品说明、脚本）**
- 可用 A/B、风格约束、长度/关键词等指标，但仍需人审
1. **数据标注 / 评审**
- 可抽样一致性、golden set，但全量正确性不易自动保证
1. **Bug triage / 方案设计**
- 结果有价值但验证不总是二值化

**适合的拍卖机制**

- scoring auction（价格 + 交付计划 + 过往信誉 + 质保条款）
- 常配合“修订轮次/退款比例/抽检规则”

## Level 3：高权限/高风险/长周期服务（High-privilege / High-stakes / Continuous）

**特征**

- 需要访问敏感系统/真实凭证（邮箱、支付、生产环境、客户数据）
- 交付往往是长期运行的服务，不是一锤子买卖
- 必须有 **最小权限 capability、审计、限额、紧急停止、法律/合规条款**
- 支付更像“合同”：按 SLA/里程碑结算，违约可罚没

**典型对象**

1. **代运营/代执行型 agent**
    - 自动化处理邮件/工单/客户沟通、投放运营、采购下单
2. **生产系统操作**
    - 部署、运维、数据管道写入、权限变更
3. **企业集成与合规相关任务**
    - 涉及审计、隐私、合规报告、财务/支付链路
4. **多 agent 协作交付**
    - 需要团队式交付与分账、责任界定

**适合的拍卖机制**

- 先“资格预审”（资质/担保/安全级别）再竞价
